# P2 架构级修复计划

## 概述

修复架构审查中识别的 4 个 P2 级别问题。按依赖顺序分 4 个 Task 执行。

---

## Task 1: Player 状态所有权 — 消除共享引用隐患

**问题**: GameSimulator、AIAgent、BeliefSystem 共享同一 Player 对象引用。simulator-day.ts 中 `target.stress = ...` 直接修改，AI 侧通过 `getPublicPlayerStates` 获取浅拷贝但嵌套对象（relations、items）仍是共享引用。

**方案**: 引入 `PlayerStateBus` — 集中式状态变更总线，所有 Player 变更通过它执行，自动发射变更事件。

### 改动文件

1. **新建 `src/lib/game/player-state-bus.ts`**
   - `PlayerStateBus` 类：维护 `players: Player[]` 作为唯一 source of truth
   - `mutatePlayer(playerId, mutation)` — 执行变更 + 发射 `StateChangeEvent`
   - `killPlayer(playerId)` — 统一的死亡处理（设 alive=false + 通知所有 Agent）
   - `changeStress(playerId, delta)` — 压力变更
   - `changeRelation(fromId, toId, delta)` — 关系变更
   - `damageItem(playerId, itemId)` — 道具损坏
   - 事件类型：`player_killed`、`stress_changed`、`relation_changed`、`item_damaged`

2. **修改 `src/lib/game/simulator-core.ts`**
   - 构造函数中创建 `PlayerStateBus` 并注入到所有 AIAgent
   - `_checkWinCondition` 从 bus 读取存活状态

3. **修改 `src/lib/game/simulator-day.ts`**（13 处 stress 变更 + 2 处 alive 变更）
   - 所有 `target.stress = clampStress(...)` → `sim.playerStateBus.changeStress(target.id, delta)`
   - `actor.alive = false; target.alive = false` → `sim.playerStateBus.killPlayer(actor.id)` + `sim.playerStateBus.killPlayer(target.id)`
   - `updateRelation` 调用改为走 bus

4. **修改 `src/lib/game/simulator-night.ts`**（3 处 alive 变更 + 3 处 damageItem）
5. **修改 `src/lib/game/simulator-vote.ts`**（1 处 alive 变更）
6. **修改 `src/lib/game/simulator-morning.ts`**（stress 恢复）
7. **修改 `src/lib/ai/ai-agent.ts`** — 构造函数接收 bus 引用
8. **修改 `src/lib/game/simulator-utils.ts`** — `getPublicPlayerStates` 从 bus 获取深拷贝

### 验证
- `bun run test` 全部通过
- `bun run build` 通过

---

## Task 2: 统一决策路径 — 意图系统驱动 + 策略执行

**问题**: 意图系统和策略系统并行运行，产生重复候选。VillagerDayStrategy 和 WerewolfCamouflageStrategy 各自做了阵营检查，与意图系统的硬约束重复。

**方案**: 让意图系统成为 "做什么" 的唯一来源，策略系统退化为 "怎么做" 的执行器。

### 改动文件

1. **修改 `src/lib/ai/strategies/engine.ts`**
   - `decide()` 中移除意图驱动补充候选（L120-159）和意图驱动评分调整（L176-205）
   - 这些逻辑移到 `IntentionManager.generateCandidates()` 中

2. **修改 `src/lib/ai/intention/intention-manager.ts`**
   - 新增 `generateCandidates(phase, context)` 方法：遍历激活意图，为每个意图的当前计划步骤生成带分数的候选

3. **修改 `src/lib/ai/strategies/day.ts`**
   - 移除 `VillagerDayStrategy` 中与意图系统重复的"预言家公布查验"逻辑（L239-269）
   - 移除 `WerewolfCamouflageStrategy` 中重复逻辑

4. **修改 `src/lib/ai/strategies/index.ts`** — 移除未使用的 `ProphetClaimStrategy`
5. **修改 `src/lib/ai/ai-agent.ts`** — 确保意图候选和策略候选在同一评分体系中

### 验证
- `bun run test` 全部通过
- 意图一致性应提高

---

## Task 3: Day Actions 插件化 — 消除 498 行 switch-case

**问题**: `simulator-day.ts` 的 `resolveDayAction` 是 498 行的巨型 switch-case。

**方案**: 参照 night actions 的插件模式，将核心 day action 迁移为独立插件。

### 改动文件

1. **新建 6 个 day action 插件** `src/lib/plugins/items/`:
   - `suspect.ts` — 怀疑（对抗检定）
   - `defend.ts` — 袒护（简单检定）
   - `accuse.ts` — 强烈指认（对抗检定）
   - `call-vote.ts` — 号召投票（简单检定）
   - `block-vote.ts` — 阻止投票（简单检定）
   - `guarantee.ts` — 担保清白（简单检定）

2. **修改 `src/lib/plugins/index.ts`** — 注册 day action 插件
3. **修改 `src/lib/game/simulator-day.ts`** — `resolveDayAction` 改为通过 `pluginRegistry.executeAction()` 调用，switch-case 缩减到 ~100 行
4. **修改 `src/lib/ai/strategies/day.ts`** — 评分逻辑迁移到对应插件的 `evaluate()`

### 验证
- `bun run test` 全部通过
- 日志输出格式与重构前一致

---

## Task 4: 事件溯源（轻量级） — 状态变更审计

**问题**: `target.alive = false` 直接 mutation，无法回放/撤销/审计。

**方案**: 在 Task 1 的 `PlayerStateBus` 基础上增加事件日志。

### 改动文件

1. **修改 `src/lib/game/player-state-bus.ts`**（Task 1 已创建）
   - 增加 `eventLog: StateEvent[]` 数组
   - 每次 mutatePlayer 自动记录事件
   - 新增 `getEventLog()` 和 `replay()` 方法

2. **新建 `src/lib/game/__tests__/player-state-bus.test.ts`**

### 验证
- 新测试全部通过

## 执行顺序

```
Task 1 (Player 状态所有权) → Task 4 (事件溯源)
Task 2 (统一决策路径) — 独立
Task 3 (Day Actions 插件化) — 独立
```

建议串行：Task 1 → Task 4 → Task 2 → Task 3
